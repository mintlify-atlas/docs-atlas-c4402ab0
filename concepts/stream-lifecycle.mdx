---
title: Stream Lifecycle
description: Understanding the states and transitions of StreamSwap streams
---

StreamSwap streams progress through six distinct states, each with specific behaviors and allowed operations. The lifecycle is time-based and deterministic, ensuring predictable behavior for all participants.

## Stream States

### Waiting

The initial state when a stream is created. In this state, the stream is not yet active and no interactions are permitted.

**Characteristics:**
- No subscriptions allowed
- No withdrawals allowed
- No spending occurs
- Stream admin can cancel the stream
- Transitions to Bootstrapping at `bootstrapping_start_time`

```rust packages/types/src/stream/stream.rs
/// Waiting status is when the stream is created. 
/// In this status, no one can interact with the stream.
Waiting,
```

### Bootstrapping

The initial participation phase where users can subscribe to the stream before distribution begins.

**Characteristics:**
- Subscriptions are **permitted**
- Withdrawals are **permitted**
- No spending occurs on either side
- No token distribution happens yet
- Participants can adjust positions freely
- Transitions to Active at `start_time`

```rust packages/types/src/stream/stream.rs
/// Bootstrapping status is when the stream is bootstrapping.
/// In this status, subscriber and withdraw are permitted. 
/// But no spending is allowed on each side.
Bootstrapping,
```

<Info>
The bootstrapping phase is ideal for participants to establish their positions before the actual distribution begins. No assets are spent during this period.
</Info>

### Active

The core operational state where token distribution and spending occur.

**Characteristics:**
- Subscriptions are **permitted**
- Withdrawals are **permitted**
- **Spending is active** on both sides
- Token distribution occurs continuously
- The `dist_index` updates as time progresses
- Price discovery happens dynamically
- Transitions to Ended at `end_time`

```rust packages/types/src/stream/stream.rs
/// Active status is when the stream is active. 
/// In this status, spending is allowed on each side.
Active,
```

#### Distribution During Active State

The distribution mechanism uses a time-based formula to calculate token releases:

```rust contracts/stream/src/stream.rs
let diff = calculate_diff(
    stream.status_info.start_time,
    stream.status_info.end_time,
    stream.status_info.last_updated,
    now,
);

if !stream.shares.is_zero() && !diff.is_zero() {
    let new_distribution_balance = stream
        .out_remaining
        .multiply_ratio(diff.numerator(), diff.denominator());
    let spent_in = stream
        .in_supply
        .multiply_ratio(diff.numerator(), diff.denominator());

    stream.spent_in += spent_in;
    stream.in_supply -= spent_in;

    if !new_distribution_balance.is_zero() {
        stream.out_remaining -= new_distribution_balance;
        stream.dist_index += Decimal256::from_ratio(new_distribution_balance, stream.shares);
        stream.current_streamed_price = Decimal256::from_ratio(spent_in, new_distribution_balance);
    }
}
```

### Ended

The stream has reached its `end_time` and awaits finalization.

**Characteristics:**
- No new subscriptions
- Subscribers can exit the stream
- Stream creator can finalize the stream
- Finalization checks if threshold was reached
- Transitions to either:
  - `Finalized(ThresholdReached)` if threshold met
  - `Finalized(ThresholdNotReached)` if threshold not met

```rust packages/types/src/stream/stream.rs
/// Ended status is when the stream is ended.
/// In this status, Subscriber can exit the stream, 
/// creator can finalize and collect accumulated in assets.
Ended,
```

### Finalized

The stream has been finalized by the creator, with two possible outcomes:

#### ThresholdReached

The stream successfully reached its minimum spending threshold.

**Outcomes:**
- Subscribers receive purchased out tokens
- Creator receives accumulated in tokens minus fees
- Exit fees are charged (from `exit_fee_percent`)
- Post-stream actions execute (pools, vesting)

```rust packages/types/src/stream/stream.rs
/// Indicates that the stream's threshold was reached.
ThresholdReached,
```

#### ThresholdNotReached

The stream failed to meet its minimum spending threshold.

**Outcomes:**
- All participants receive **full refunds**
- No fees are charged
- Creator gets all out tokens back
- Subscribers get all in tokens back

```rust packages/types/src/stream/stream.rs
/// Indicates that the stream's threshold was not reached.
ThresholdNotReached,
```

<Warning>
When a stream is finalized with ThresholdNotReached, all participants are fully refunded. This protects both creators and subscribers from unsuccessful streams.
</Warning>

### Cancelled

The stream was cancelled before or during execution.

**Characteristics:**
- Can only be cancelled by stream admin during Waiting state
- Can be cancelled by protocol admin at any time (emergency)
- All participants receive **full refunds**
- Subscribers get all in tokens back
- Creator gets all out tokens back
- No fees charged

```rust packages/types/src/stream/stream.rs
/// Cancelled status is when the stream is cancelled.
/// In this status, Subscriber can exit the stream and collect full in assets.
/// Creator can collect full out assets.
Cancelled,
```

## State Transition Diagram

The state transitions are time-based and follow this flow:

```
Waiting
  ↓ (at bootstrapping_start_time)
Bootstrapping
  ↓ (at start_time)
Active
  ↓ (at end_time)
Ended
  ↓ (on finalize call)
Finalized(ThresholdReached) OR Finalized(ThresholdNotReached)

* Cancelled can occur from Waiting (by stream_admin) or any state (by protocol_admin)
```

### Automatic State Synchronization

Stream status is automatically synchronized based on block time:

```rust contracts/stream/src/stream.rs
pub fn sync_stream_status(stream: &mut StreamState, now: Timestamp) {
    if matches!(
        stream.status_info.status,
        Status::Cancelled
            | Status::Finalized(FinalizedStatus::ThresholdNotReached)
            | Status::Finalized(FinalizedStatus::ThresholdReached)
    ) {
        return;
    }
    stream.status_info.status = match now {
        _ if now < stream.status_info.bootstrapping_start_time => Status::Waiting,
        _ if now >= stream.status_info.bootstrapping_start_time
            && now < stream.status_info.start_time =>
        {
            Status::Bootstrapping
        }
        _ if now >= stream.status_info.start_time && now < stream.status_info.end_time => {
            Status::Active
        }
        _ if now >= stream.status_info.end_time => Status::Ended,
        _ => stream.status_info.status.clone(),
    };
}
```

## Threshold Mechanism

Streams can specify an optional spending threshold:

```rust packages/types/src/stream/stream.rs
pub fn check_threshold(&self) -> bool {
    match self.threshold {
        Some(threshold) => self.spent_in >= threshold,
        None => true,
    }
}
```

- **With threshold**: Stream must reach minimum `spent_in` amount to be considered successful
- **Without threshold**: Stream always succeeds when ended

This protects creators from launching tokens with insufficient participation.

## Duration Requirements

The protocol enforces minimum durations for each phase:

```rust packages/types/src/controller/params.rs
// Minimum time of a stream end_time - start_time
pub min_stream_duration: u64,
// Minimum time of bootstrapping status
pub min_bootstrapping_duration: u64,
// Minimum time of waiting status
pub min_waiting_duration: u64,
```

These parameters ensure streams have adequate time for participation and prevent gaming.
