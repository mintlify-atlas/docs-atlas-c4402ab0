# Architecture Overview

StreamSwap's architecture is designed to support continuous and fair execution of token distribution through a modular, two-contract system.

## System Design

StreamSwap employs a factory pattern where the Controller Contract creates and manages multiple Stream Contracts. This design provides:

- **Separation of concerns**: Protocol-level operations vs. stream-specific logic
- **Scalability**: Each stream operates independently
- **Upgradeability**: Contracts can be migrated without affecting existing streams
- **Governance**: Centralized parameter management with decentralized execution

## Core Contracts

### Controller Contract

The Controller Contract serves as the protocol's central coordinator and factory.

<Accordion title="Key Responsibilities">
  - **Stream Creation**: Instantiates new stream contracts using the stored code ID
  - **Parameter Management**: Maintains protocol-wide configuration (fees, durations, accepted denoms)
  - **Fee Collection**: Collects stream creation fees and manages the fee collector address
  - **Governance Interface**: Provides admin functions for protocol updates
  - **Registry**: Tracks all created streams with unique IDs
  - **Validation**: Enforces minimum durations and validates stream configuration
</Accordion>

#### Controller State

```rust
// Core parameters stored in the Controller
PARAMS: Params {
    stream_contract_code_id: u64,      // Code ID for stream instantiation
    vesting_code_id: u64,               // Code ID for vesting contracts
    protocol_admin: Addr,               // Admin address for governance
    fee_collector: Addr,                // Address receiving fees
    stream_creation_fee: Coin,          // Fee required to create a stream
    exit_fee_percent: Decimal256,       // Exit fee percentage (0-100%)
    accepted_in_denoms: Vec<String>,    // Whitelisted input denominations
    min_waiting_duration: u64,          // Minimum waiting phase duration
    min_bootstrapping_duration: u64,    // Minimum bootstrapping phase duration
    min_stream_duration: u64,           // Minimum active stream duration
    tos_version: String,                // Terms of service version
}

LAST_STREAM_ID: u64                     // Auto-incrementing stream ID
STREAMS: Map<u64, Addr>                 // Maps stream ID to contract address
FREEZESTATE: bool                       // Emergency freeze flag
```

### Stream Contract

Each Stream Contract manages an individual token swap stream with its own lifecycle and participants.

<Accordion title="Key Responsibilities">
  - **Subscription Management**: Handles user subscriptions with share calculations
  - **Continuous Distribution**: Distributes tokens proportionally over time
  - **Price Calculations**: Computes average and streamed prices in real-time
  - **Position Tracking**: Maintains individual participant positions and spent amounts
  - **Withdrawal Logic**: Processes withdrawals with proportional share reduction
  - **Exit Processing**: Handles stream exits and refunds after completion
  - **Finalization**: Executes treasury collection with exit fees
  - **Post-Stream Actions**: Creates vesting schedules and liquidity pools
  - **Threshold Validation**: Checks minimum raise requirements
</Accordion>

#### Stream State

```rust
// Stream state tracked throughout lifecycle
STREAM_STATE: StreamState {
    // Token configuration
    out_asset: Coin,                    // Token being distributed
    out_remaining: Uint256,             // Remaining tokens to distribute
    in_denom: String,                   // Input token denomination
    in_supply: Uint256,                 // Current subscribed input amount
    spent_in: Uint256,                  // Total spent input tokens
    
    // Share tracking
    shares: Uint256,                    // Total shares in the stream
    dist_index: Decimal256,             // Distribution index for calculations
    
    // Pricing
    current_streamed_price: Decimal256, // Latest streamed price
    
    // Timing and status
    status_info: StatusInfo {
        status: Status,                 // Current lifecycle status
        bootstrapping_start_time: Timestamp,
        start_time: Timestamp,
        end_time: Timestamp,
        last_updated: Timestamp,
    },
    
    // Optional configurations
    threshold: Option<Uint256>,         // Minimum raise threshold
}

// Individual participant positions
POSITIONS: Map<Addr, Position> {
    owner: Addr,
    in_balance: Uint256,                // Unspent input balance
    shares: Uint256,                    // Position shares
    index: Decimal256,                  // Distribution index snapshot
    spent: Uint256,                     // Spent input amount
    purchased: Uint256,                 // Purchased output amount
}

// Stream metadata
STREAM_INFO: StreamInfo {
    name: String,
    url: Option<String>,
    treasury: Addr,
    stream_admin: Addr,
}

// Post-stream configurations
POST_STREAM: PostStreamActions {
    pool_config: Option<PoolConfig>,
    subscriber_vesting: Option<VestingConfig>,
    creator_vesting: Option<VestingConfig>,
}
```

## Stream Lifecycle States

Streams progress through distinct phases with specific participant capabilities:

<Steps>
  <Step title="Waiting">
    **Status**: Stream created but not yet active
    
    **Allowed Actions**:
    - Cancel (stream admin)
    - Update stream parameters (stream admin)
    
    **Restrictions**: No subscriptions or distributions
  </Step>
  
  <Step title="Bootstrapping">
    **Status**: Subscription phase before distribution begins
    
    **Allowed Actions**:
    - Subscribe (users)
    - Withdraw (users)
    - Cancel (stream admin)
    
    **Restrictions**: No spending occurs, tokens are not distributed yet
  </Step>
  
  <Step title="Active">
    **Status**: Stream is live and distributing tokens
    
    **Allowed Actions**:
    - Subscribe (users)
    - Withdraw (users)
    - Query positions and prices
    
    **Key Behavior**: Continuous distribution based on elapsed time and total shares
  </Step>
  
  <Step title="Ended">
    **Status**: Stream duration completed
    
    **Allowed Actions**:
    - ExitStream (users)
    - FinalizeStream (treasury)
    - Cancel (stream admin)
    
    **Restrictions**: No new subscriptions or withdrawals
  </Step>
  
  <Step title="Finalized">
    **Status**: Treasury has collected proceeds
    
    **Allowed Actions**:
    - ExitStream (users)
    - Query final results
    
    **Post-Actions**: Vesting schedules created, pools established (if configured)
  </Step>
  
  <Step title="Cancelled">
    **Status**: Stream cancelled by admin
    
    **Allowed Actions**:
    - ExitStream (users) - Full refund
    - Treasury can reclaim unsold tokens
    
    **Behavior**: All participants receive full refunds of unspent amounts
  </Step>
</Steps>

## Distribution Algorithm

StreamSwap's continuous distribution algorithm is the core innovation:

### Time-Based Distribution

```rust
// Calculate distribution portion based on elapsed time
let time_elapsed = now - last_updated;
let total_duration = end_time - start_time;
let distribution_ratio = time_elapsed / total_duration;

// Distribute tokens proportionally
let tokens_to_distribute = out_remaining * distribution_ratio;
let input_to_spend = in_supply * distribution_ratio;

// Update distribution index for all participants
dist_index += tokens_to_distribute / total_shares;
```

### Share Calculations

```rust
// New subscription: shares based on current rate
if shares == 0 {
    new_shares = amount_in;  // First subscriber gets 1:1
} else {
    new_shares = (total_shares * amount_in) / in_supply;
}

// Individual distribution: based on shares and index
user_purchased = user_shares * (current_dist_index - user_index);
```

## Key Design Patterns

### Circuit Breaker

The Controller implements a freeze mechanism to pause stream creation in emergencies without affecting existing streams.

### Distribution Index

Using a distribution index pattern allows efficient tracking of per-user distributions without iterating over all users.

### State Synchronization

Stream state is lazily updated on user interactions, ensuring efficient gas usage while maintaining accurate distributions.

### Factory Pattern

Controller acts as a factory, creating isolated stream instances with consistent parameters and interfaces.

## Security Considerations

<Warning>
  **Key Security Features**:
  - Threshold validation prevents finalization below minimum raise
  - Exit fees discourage late-stage manipulation
  - Admin controls separated from treasury operations
  - Circuit breaker for emergency situations
  - Comprehensive validation of timing parameters
</Warning>
